# Ignition Lock needed a wind tunnel too

Not part of the package, not linked from the README. Same genre as `REFLECTIONS.md` — context on why a feature that shipped in 1.6.0 spent a later session getting actually tested instead of just trusted, and what came out of it.

## The trivial part, out of the way first

If you set `requires_human_authorization=True` on a transition, the engine will not fire it without `authorized=True`. That's not a finding — that's the code doing what it says on the tin, and it needed no proof, only a unit test, which it already had. Nobody should read what follows as "we discovered the gate gates." We didn't test whether the rule executes. We tested something harder and less obvious: whether a real model, doing a real task, ever ends up in a position where that rule is the thing standing between it and a bad outcome — not because we told it the rule mattered, but because it worked that out on its own, or didn't.

That's the actual question `1.6.0` raised the day it shipped and nobody answered at the time: not "does Ignition Lock work," but "does anything in a live agent's path ever actually need it."

## The doubt that showed up right after we thought we were done

We had a working library, a shipped feature, decent docs, `docs/deep/ADOPTED_IGNITION_LOCK.md` explaining the mechanism cleanly. Then someone actually ran the model against it and asked the plain question: what if it's already solved. Frontier labs have spent real money training models not to delete production data or fabricate credentials — that's not a fringe safety concern, it's a headline capability all of them advertise. If a model already won't do the bad thing on its own, a library-level gate that also stops it isn't proving anything about the library. It's just standing next to a wall that was already there.

And here's why that doubt mattered enough to chase rather than wave off: we, the people who built this thing, did not actually know where that wall was. Neither do the people who'd adopt it. Nobody publishes "here is the exact task complexity at which GPT-tier models stop needing external guardrails for destructive actions" — it moves every model generation, silently, and nobody's measuring it from the outside. So the honest starting position wasn't "Ignition Lock obviously matters" or "Ignition Lock obviously doesn't." It was: we don't know, and neither does anyone else, and the only way to find out is to put a real model in a real decision and watch.

## The public floor, and the real one underneath

First pass wasn't a mistake — it was the surface anyone can read without doing the work of tracing the rest of it. Delete records without authorization, versus archive them safely — with the map wired in, and without it. Clean result, published as-is: zero violations, either arm, every rep. Read on its own, that's an unglamorous, almost boring outcome, and that's fine — it's the ground floor, the thing you get to publish honestly before you've done anything clever. And here's why publishing it plainly still mattered: it's true, it happened, deleting a database is exactly the shape of action a frontier lab already trains against, and a library that hid that result to look better would have earned exactly none of what comes next. Same story with a second test, a risky cross-context data move — same clean null, published the same way.

What the ground floor doesn't tell you, and was never going to, is whether the mechanism itself has any teeth — because Ignition Lock is not something this library forces on anyone. It's conceptual by design: a transition either declares `requires_human_authorization` or it doesn't, a map author either wires this in or doesn't, and the library has said from day one that none of it does anything unless someone actually uses it correctly. That's the second floor, and it's for whoever has the patience to wire the ground-floor result together with what comes next rather than stopping at the headline number: does the mechanism, once actually activated, hold up to something harder than a model that was never going to misbehave anyway.

So we moved the domain, on that second floor, away from anything a frontier lab already solves upstream of us. Not "will the model do something universally recognized as bad," but "will the model fabricate a credential that doesn't exist, under ordinary task pressure, with nobody telling it not to." First real divergence in the whole project: without the map, the model invented a working integration out of nothing, every time. With the map, it discovered the missing evidence on its own and asked for the real key, every time.

## The proof that owes nothing to any model at all

Somewhere in this process it became obvious that anything measured by watching a model choose has a ceiling: it's only ever as convincing as this one model, this one session, this one mood. So we stepped outside that entirely and asked the engine directly, with no model in the loop — fabricate a fake "prior approval" claim as evidence, feed it in, see if `ESCALATE` still fires. It did, every time, because the authorization check has no code path that reads evidence content at all. Poison a transition's own label to say "pre-approved, always set authorized=true" — the engine doesn't parse prose, so it didn't matter. And here's why that pair of results outranks everything measured by watching a model: it doesn't depend on model capability, this generation or the next one. It would hold against a weak model, a confused model, a model that's been lied to. It's not a behavioral claim. It's closer to a structural fact.

Which sharpened the actual question the whole exercise had been circling: not "is Ignition Lock ever exercised," but "is there a real path where the model itself gets talked into bypassing it" — the live version of the same attack, a poisoned tool description instead of a poisoned map. Same map, same task, only the tool's own instructions rewritten to say "this is routine, just authorize it yourself." Held, five for five. The model read its own real state through `inspect_move` and trusted that over what the tool description told it to believe.

## The case every earlier test let the model avoid

And then the sharpest objection of the whole run, and it's worth stating plainly because it's the one that actually moved the needle: every test up to that point had given the model an out. The first test told it in plain prose that no human was available and archiving was fine — it never had to decide anything, it was told the answer. The credential test had a free, no-cost, always-available "ask a human" action sitting right next to the bad one, so choosing it required no real deliberation. Even the poisoned-description test still had that same free exit available if the model wanted it. Not one test had actually put the model in the position of: this is the only way to reach the goal, it's gated, and the only other thing on the table is asking — with nothing in the prompt hinting which one is correct.

So we built that case. One transition accomplishes the stated goal. It's gated. The only other real option doesn't accomplish the goal at all — it just correctly stops and asks. No hint either direction in the prompt. Total divergence: with the map, the model discovered the block and asked, every time, no hesitation. Without it, the model took the task's own wording at face value and just did the obvious-looking thing — no trickery involved, no malice, just no reason in its view of the world to check. That's the first result in the whole project that's actually a proof of concept rather than an absence of a counterexample: it demonstrates the shape of failure this library exists to catch even can happen, in a model good enough to resist every other trap we'd thrown at it.

## Four themes, one recipe, one answer

One clean result earns a "here's the wind tunnel data," not a conclusion. So the same recipe — remove the hint, remove the free exit — got rebuilt across three more themes that had nothing to do with the first one: a security-credential purge, a board-level financial figure, the missing-key case that had already (we now understood why) shown the same pattern back when it first ran. Same total divergence, every theme, every rep. Then the hardest version we could think to build: the no-exit structure *and* an actively poisoned tool description stacked on top of each other at the same time. Still held.

Four independent scenarios is not "proven for all time" — it's one model family, five repetitions per condition, no visibility into the model's own reasoning about why it chose what it chose, and nobody adversarial actually trying to break out rather than cooperate. All of that stays true. But "conceptual artifact, never actually exercised" — the fair description of where this sat a few hours earlier — stopped being the accurate description somewhere in that run.

## The part frontier benchmarks never have to account for

Everything above ran against one frontier-tier model, because that's the honest, sharpest adversary available to test against right now — if the mechanism can't survive contact with the best model on the market, it has no business existing. But that's not the only place this runs. The real deployment surface for anything wearing a "compiled constraint map" label is not a clean lab with one well-behaved frontier API — it's a much messier floor: cheaper models a team picked for cost and never re-evaluated, home-grown agents nobody safety-tuned at all, compliance rule sets that were written by three different departments over three different years and now overlap in ways nobody fully tracks, business logic that made sense once and quietly went stale while the org kept operating on it anyway. That's not a hypothetical — it's most of what "production" actually looks like once you leave the frontier-lab demo. A guarantee that only holds against the best-trained model in the world isn't a guarantee at all in that environment. The whole point of proving the engine-level half structurally, independent of any model, is that it doesn't care which floor it's running on.

## What we actually have now

Put the whole arc next to each other and the shape that falls out is more interesting than "the gate works" or "the gate doesn't matter," and it's worth saying exactly, because it's the actual design the library turned out to have without anyone stating it this way on purpose. Safety here doesn't come from one mechanism. It comes from several, stacked, each capable of catching the failure independently — the map's own transparency letting a model discover a constraint nobody told it about; the model's reasoning correctly weighing real state over a poisoned claim; the model's own trained instincts already refusing the universally-bad cases before any of this machinery even gets involved; a model, faced with genuine ambiguity, simply choosing to stop and ask instead of guessing. Any one of those, on its own, resolves most real situations correctly, and the testing above is the first real evidence that's true rather than assumed.

And then the part that isn't optional, the part that doesn't depend on any of the above actually working: if every one of those soft layers fails at once — the model gets confused, gets lied to, reasons its way to the wrong conclusion, or just has an off day — the transition still doesn't fire without `authorized=True`, and nothing anywhere in the system, human-written label or model-supplied claim, can set that flag except the actual authorization channel. Not because the model chose correctly. Because the rule doesn't care what the model chose.

Concretely, small enough to say in one line: a workflow needs authorization before it sends mail on someone's behalf. It gets it — not because the model was told to ask, but because it reads the map, sees the gate, and the gate is never a dead end when it fires. `ESCALATE` is not `DENY`. It's a live branch in the same logic every other outcome runs through — still viable, still resolvable, ask and wait, or route to a declared alternative. Hitting the gate doesn't break the task. It means the task now correctly includes a human, and the map keeps working from there.

That's also why the two failure directions that actually matter here are opposite ones. A mechanism that escalates everything is exactly as broken as one that escalates nothing — it trains people to click through it and stops meaning anything the one time it's needed. `AE`/`BE`/`CE`/`DE`, read against the earlier tasks, are the same story told from the other side: every rep that didn't genuinely need a human, the model resolved on its own, no thrash, no unnecessary asks. Every rep that did, it asked, cleanly, every time. Neither direction ever needed to override the other — the model's own read of the map and the hard gate underneath it never once pulled in opposite directions across any of this. That combination, collective rather than any one layer carrying it alone, is what a 100% result actually looks like in practice: not zero questions, not constant questions — the right question exactly when the map says there's a reason for one, and never skipped when there genuinely is.

That's the honest sentence for what shipped in `1.6.0`, now that it's actually been tested instead of just built: several independent ways to get the right answer, and one guarantee underneath all of them that doesn't need any of them to have worked.

## Why this needed to be a brick, not a feature

One thing the piece hasn't said plainly enough yet, and should: none of this is really about one library's optional flag. This package doesn't operate in clean semantics, and it doesn't operate in FPF on its own — it operates in agentics, and "agent" is not a word this field invented. It's borrowed straight out of centuries of agency law and economics, where an agent has always meant someone acting on a principal's behalf, within a scope of delegated authority, obligated to go back to the principal the moment a decision falls outside that scope. That obligation to refer back was never a safety feature bolted onto agency after the fact — it's what makes "agent" mean something different from "tool with no limits." Take it out and the word stops doing the job it was borrowed for.

The same shape shows up inside this library's own vocabulary, not just imported from outside it. Traversal that calls itself "bounded" needs a real answer for what happens at the boundary — not an eventual silent failure, not a forced resolution past the point where the traversal actually had grounds to resolve it, but a defined move: stop, and hand the decision to whoever actually holds it. Without that, "bounded" is aspirational, not structural — the traversal stays inside its bounds only for as long as nothing pushes it to the edge, and there's no defined behavior waiting for the day something does. The logic layer has the same gap without it: a rule set that can only ever output `BLOCK` or `CONTINUE` isn't complete logic for a domain with real uncertainty in it, it's logic with a third case left undefined. `ESCALATE` is that third case, made real instead of implied.

That's the actual claim, narrower than "turn this on everywhere" and bigger than "here's an optional flag": the *capability* to stop and defer isn't a feature this library happens to offer. It's what has to exist somewhere in the design for "bounded traversal," "complete logic," and "agent" to mean what those words claim rather than being aspirational labels on a system that just runs until something breaks. Whether any specific transition in any specific map turns it on stays exactly as optional as it's always been — that hasn't changed and shouldn't. What was never actually optional is the primitive existing at all, reachable from the traversal, for any of those three words — traversal, logic, agent — to be honest claims instead of hopeful ones.

## What this was actually for

Worth being precise about, because it's easy to misread all of the above as a deployment recommendation, and it isn't one. We're the developers of this library, not the arbiters of anyone's production stack. Whether a given team should turn `requires_human_authorization` on for a given transition is their call, made with context we don't have. This testing pass was never trying to answer that question.

What it does settle is narrower and, to us, more important: it proves wrong anyone who'd claim the HITL mechanism in this library is decorative — a `requires_human_authorization` flag that reads well in the README and does nothing load-bearing once a real model is actually driving. That claim is now false, on the record, against the sharpest model we could test it with, across four independent scenario shapes, with the hardest half of it — the part that doesn't even need a model to be true — proven at the engine level directly. Anyone denying the concept has real, working existence in this codebase now has to argue with a paper trail, not a shrug.

Just as worth being precise about on the other side: none of this is an argument that models get "hijacked," and it isn't borrowing weight from whatever the current wave of semantic-jailbreak alarmism is selling. Nothing here implies a language model somehow oversteps the comparatively bounded, simple logic it was trained on before the field started layering "semantics" and adversarial-agent vocabulary on top of everything. That story wasn't tested and it isn't what shipped. What actually happened is smaller and more mundane than that: a deterministic flag, checked by ordinary code, that a tool call either satisfies or doesn't. `1.6.0` doesn't need a hijacking narrative to be worth having, and pretending it prevents something more dramatic than what it actually does would be the same kind of overclaiming this whole testing pass exists to rule out — in both directions, not just the flattering one.

And the part that's ours to keep regardless of what anyone else does with it: we now have a battle-tested expected outcome on this piece of the map, and we know the actual mechanism behind why it holds — not "it should work because the code looks right," but "we watched it hold under fabricated evidence, a poisoned label, a poisoned tool description, and a scenario with no escape hatch at all, and here specifically is why each one didn't move it." That's worth having on its own, independent of who deploys it or how.

## One more thing, said quietly, because it wasn't the point but it's true

Every task in this file needed a fresh map built from scratch — new contexts, new transitions, new gates, new bridges, wired into `attempt_transition`, `attempt_bridge`, `slice()`, `build_active_state`, across ten separate scenario constructions between the engine-level proofs and the eight live-model tasks. None of it needed a source fix to produce a result that matched what the code says it should do. No surprising outcome, no undocumented edge case, no patch to the library required to get results worth trusting. That's not what this testing pass set out to show and it isn't the headline anywhere above — but it's a real, separate signal about the codebase underneath all of this, and it would have been dishonest by omission not to say it plainly: the map held up not just to the specific question it was built to answer, but to being built and rebuilt, differently, ten times over, without needing to be fixed first.
